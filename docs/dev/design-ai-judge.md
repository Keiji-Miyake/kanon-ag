# AI Judge 詳細設計

## 1. はじめに

### 1.1 目的と背景
本ドキュメントは、ロードマップのマイルストーン2「AI Judge」の詳細設計を定義します。
自律型エージェントのオーケストレーションにおいて、ループの停滞や不確実な状況が発生した場合に、現状のコンテキストを客観的に評価する仕組みが必要です。本設計では、独立したスーパーバイザーエージェントである「AI Judge」を導入し、「進捗の有無」や「成功の可能性」を判定して、タスクのABORT（中断）、継続、またはユーザー介入の判断を自動化します。

### 1.2 AI Judge の役割
- これまでの実行履歴（コンテキスト）から客観的に進捗状況を評価する。
- タスクが成功する見込みがあるかを判定する。
- 実行の継続、ABORT、またはユーザーへの確認（介入）を決定する。

## 2. アーキテクチャ概要

クリーンアーキテクチャの原則に基づき、AI Judge機能の責務を各層に分離します。

### 2.1 Domain Layer (エンティティ)
- **`EvaluationResult`**: AI Judgeの判定結果を表す値オブジェクト。
  - `status`: `CONTINUE` (継続), `ABORT` (中断), `ESCALATE` (ユーザー介入)
  - `reason`: 判定の理由
  - `confidenceScore`: 判定の確信度 (0.0 - 1.0)

### 2.2 Use Case Layer (アプリケーションロジック)
- **`EvaluateTaskProgressUseCase`**: 現在のコンテキスト（実行履歴、エラー履歴、ループ回数など）を受け取り、AI Judge（外部サービス）への問い合わせをオーケストレーションして判定結果を返す。

### 2.3 Infrastructure Layer
- **`AiJudgeClient`**: 独立したAIモデル（LLM）に対して、客観的評価を促すシステムプロンプトと共にコンテキストを送信し、判定結果を受け取るアダプター。

### 2.4 Interface Adapters (Orchestration)
- **`OrchestrationService`**: メインのオーケストレーションループ内で、特定のトリガー（例: Watchdogによる停滞検知など）に反応して `EvaluateTaskProgressUseCase` を呼び出し、結果に基づいてループ制御を変更する。

## 3. 判定基準の定義

AI Judgeは以下の基準に従って状態を判定し、`status` を決定します。

### 3.1 ABORT（中断）すべき場合
- **無限ループの確信**: 同じエラーや同じ修正アプローチが N 回（例: 3回）以上繰り返されており、新たな視点やアプローチが見られない。
- **解決不可能なブロッカー**: 必要な外部リソース（API制限、認証情報不足など）が完全に欠如しており、エージェントの権限や能力では解決できない。
- **目標からの逸脱**: エージェントのアクションが本来のタスク目標から大きく逸脱し、修復不可能な状態に陥っている。

### 3.2 継続すべき場合
- **着実な進捗**: エラーは発生しているものの、毎回異なるアプローチを試みており、段階的に解決に向かっている形跡がある。
- **一時的な障害**: ネットワークエラーやタイミングによる不具合など、再試行によって解決する可能性が高い問題。

### 3.3 ESCALATE（ユーザー介入）すべき場合
- **仕様の不確実性**: タスクの要件が曖昧であり、複数の解釈が存在するため、どちらの方向に進むべきかエージェント自身で決定できない。
- **破壊的変更の必要性**: 進行のために、広範囲なリファクタリングやアーキテクチャの変更など、ユーザーの承認が必要な重大な判断が求められる場合。
- **確信度の低いABORT**: 失敗の可能性が高いが、AI Judge自身も明確な確信が持てない（`confidenceScore` が低い）場合。

## 4. `OrchestrationService` への統合方法

### 4.1 AI Watchdog との違い
- **AI Watchdog**: 主に「メトリクス（トークン消費量、ループ回数、経過時間）」ベースの受動的・定量的な監視役。閾値を超えた場合に警告を発する（トリガーを引く役割）。
- **AI Judge**: 定性的な「スーパーバイザー」。Watchdogが異常を検知した際や、エージェントが「迷い」を表明した際に起動し、過去のコンテキスト（会話履歴や実行ログ）を深く読み込み、意味論的に進捗を評価する役割。

### 4.2 統合フロー
1. `OrchestrationService` のメインループ実行中。
2. `AI Watchdog` が停滞（例: 同じステップで3回のエラー）を検知。
3. `OrchestrationService` は実行を一時停止し、`EvaluateTaskProgressUseCase` を起動（AI Judgeへの問い合わせ）。
4. AI Judge がコンテキストを分析し、`EvaluationResult` を返す。
5. `OrchestrationService` の処理:
   - `CONTINUE`: Watchdogのカウンターをリセットし、エージェントにAI Judgeのアドバイス（別のアプローチの提案など）を渡してループを再開。
   - `ABORT`: オーケストレーションを即座に終了し、失敗の理由をログに出力してユーザーに報告。
   - `ESCALATE`: 実行を停止し、ユーザーに介入プロンプトを提示して回答を待機。

## 5. ユーザーへの介入プロンプト

AI Judgeが `ESCALATE` を判定した場合、ユーザーに以下のようなプロンプトを提示して判断を仰ぎます。

### 5.1 介入が必要な場合の判定（AI Judgeの出力要件）
AI JudgeはESCALATEを返す際、必ず以下の情報を出力しなければなりません。
- **現状のサマリー**: 何が起きていて、なぜ行き詰まっているのかの客観的要約。
- **課題の核心**: どの点が不確実なのか（要件の不足、技術的制約、権限の問題など）。
- **選択肢**: ユーザーが選ぶべき具体的なアクション案（A案、B案、または手動での修正など）。

### 5.2 プロンプトの設計（CLI表示例）
```text
🚨 AI Judgeからのエスカレーション: 進行にユーザーの判断が必要です。

【現状のサマリー】
データベースのスキーママイグレーションにおいて、既存のデータと競合する制約違反が発生し、3回の自動修復を試みましたが解決しませんでした。

【課題の核心】
既存の不正データを自動的に削除してよいか（破壊的変更）、あるいは仕様を変更して制約を緩和すべきかの判断がつきません。

【選択肢】
1. 既存の不正データを削除してマイグレーションを続行する
2. マイグレーションを中止し、タスクをABORTする
3. 制約を緩和する方針でエージェントに再試行させる

選択してください (1/2/3)、または具体的な指示をテキストで入力してください:
```

## 6. 実装計画

1. **ドメイン・ユースケースの定義**: 
   - `EvaluationResult` エンティティの作成
   - `EvaluateTaskProgressUseCase` インターフェースと実装の作成
2. **AI Judge クライアントの実装**: 
   - `Infrastructure` 側にLLMを使用した独立したプロンプト評価ロジックを実装。タスク実行用のエージェントとは別の、客観的評価を目的としたシステムプロンプトを適用する。
3. **OrchestrationService の改修**: 
   - AI Watchdog の検知ロジックと連動させ、必要に応じて AI Judge を呼び出す割り込み処理（Interruptロジック）を実装する。
4. **ユーザーインターフェースの拡充**: 
   - `ESCALATE` 時のプロンプト表示と、ユーザー入力の受け付け、およびその回答コンテキストを付与してループを再開する仕組みの実装。
