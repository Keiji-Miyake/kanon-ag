# 概要
Kanon のワークフロー定義を `pipeline.json` から、より柔軟な **Score (楽譜)** と **Passage (楽節)** の構造へ移行するための、第一段階の基礎を構築する。エージェントが返す JSON に基づいた遷移制御と、ハッシュによる停滞検知（Loop Watchdog）を導入する。

# 機能要件
1. **Score & Passage 構造の導入**:
   - `pipeline.json` に代わる新しい定義形式 `score.json` (または `kanon-score.json`) のプロトタイプを導入。
   - 各実行ステップを **Passage** と呼び、従来の `Phase` よりも柔軟に定義できるようにする。
2. **JSONベースの遷移制御**:
   - Passage の出力に含まれる特定の JSON コードブロック（例: `json:passage-result`）を解析し、`next_passage` フィールドの値に基づいて次のステップを決定する。
3. **Loop Watchdog (ハッシュによる停滞検知)**:
   - 各 Passage の出力全体の SHA-256 ハッシュを計算。
   - 3回連続で同一ハッシュ（出力に変化なし）の場合、AIが停滞しているとみなし、ABORT する。

# 非機能要件
- 既存の `pipeline.json` との互換性を考慮しつつ、新しい `RuleEngine` サービスを独立したモジュールとして設計すること。
- テスト駆動開発 (TDD) に従い、各判定ロジックのユニットテストを先行して実装すること。

# 受け入れ基準
- `score.json` に定義された Passages が、エージェントの出力 JSON に応じて正しく遷移すること。
- 出力が 3回変化しない場合に、正しくループを検知して停止すること。

# スコープ外
- 並列実行（Magi方式）やプロンプトの完全なモジュール化（Milestone 2以降の項目）。
