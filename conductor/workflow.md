# プロジェクトワークフロー

## 指針

1. **計画が唯一の真実:** すべての作業は `plan.md` で追跡されなければなりません。
2. **技術スタックは意図的:** 技術スタックへの変更は、実装の *前に* `tech-stack.md` に文書化されなければなりません。
3. **テスト駆動開発 (TDD):** 機能を実装する前にユニットテストを記述してください。
4. **高いコードカバレッジ:** すべてのモジュールで 80% 以上のコードカバレッジを目指してください。
5. **ユーザーエクスペリエンス優先:** すべての決定はユーザーエクスペリエンスを最優先事項とする必要があります。
6. **非対話的かつ CI 対応:** 非対話的なコマンドを優先してください。ウォッチモードのツール（テスト、リンターなど）には `CI=true` を使用して、1回だけ実行されるようにしてください。

## タスクワークフロー

すべてのタスクは厳格なライフサイクルに従います。

### 標準タスクワークフロー

1. **タスクの選択:** `plan.md` から次に利用可能なタスクを順番に選択します。

2. **進行中にマーク:** 作業を開始する前に、`plan.md` を編集してタスクを `[ ]` から `[~]` に変更します。

3. **失敗するテストの作成 (Redフェーズ):**
   - 機能またはバグ修正のための新しいテストファイルを作成します。
   - タスクの期待される動作と受け入れ基準を明確に定義する1つ以上のユニットテストを記述します。
   - **重要:** テストを実行し、期待通りに失敗することを確認します。これが TDD の "Red" フェーズです。テストが失敗するまで次に進まないでください。

4. **テストに合格するための実装 (Greenフェーズ):**
   - 失敗しているテストに合格するために必要な最小限のアプリケーションコードを記述します。
   - テストスイートを再度実行し、すべてのテストが合格することを確認します。これが "Green" フェーズです。

5. **リファクタリング (推奨):**
   - 合格したテストの安全性を確保した上で、実装コードとテストコードをリファクタリングして、外部の動作を変えずに明快さを向上させ、重複を排除し、パフォーマンスを向上させます。
   - リファクタリング後、テストを再実行して合格し続けていることを確認します。

6. **カバレッジの検証:** プロジェクトで選択されたツールを使用してカバレッジレポートを実行します。例：
   ```bash
   npm test -- --coverage
   ```
   ターゲット：新しいコードに対して 80% 以上のカバレッジ。

7. **逸脱の文書化:** 実装が技術スタックと異なる場合：
   - 実装を **停止** します。
   - 新しい設計で `tech-stack.md` を更新します。
   - 変更を説明する日付付きのメモを追加します。
   - 実装を再開します。

8. **コード変更のコミット:**
   - タスクに関連するすべてのコード変更をステージングします。
   - 明確で簡潔なコミットメッセージを提案します（例：`feat(ui): Create basic HTML structure for calculator`）。
   - コミットを実行します。

9. **Git Notes によるタスクサマリーの添付:**
   - **ステップ 9.1: コミットハッシュの取得:** 完了したばかりのコミットのハッシュを取得します (`git log -1 --format="%H"`)。
   - **ステップ 9.2: ノート内容のドラフト:** 完了したタスクの詳細なサマリーを作成します。これには、タスク名、変更の概要、作成/変更されたすべてのファイルのリスト、および変更の核心となる「なぜ」を含める必要があります。
   - **ステップ 9.3: ノートの添付:** `git notes` コマンドを使用して、サマリーをコミットに添付します。
     ```bash
     git notes add -m "<note content>" <commit_hash>
     ```

10. **タスクコミット SHA の記録:**
    - **ステップ 10.1: 計画の更新:** `plan.md` を読み、完了したタスクの行を見つけ、ステータスを `[~]` から `[x]` に更新し、完了したばかりのコミットハッシュの最初の7文字を追記します。
    - **ステップ 10.2: 計画の書き込み:** 更新された内容を `plan.md` に書き戻します。

11. **計画更新のコミット:**
    - **アクション:** 変更された `plan.md` ファイルをステージングします。
    - **アクション:** 説明的なメッセージとともにこの変更をコミットします（例：`conductor(plan): Mark task 'Create user model' as complete`）。

### フェーズ完了の検証とチェックポイントプロトコル

**トリガー:** このプロトコルは、`plan.md` 内のフェーズを締めくくるタスクが完了した直後に実行されます。

1.  **プロトコル開始のアナウンス:** フェーズが完了し、検証およびチェックポイントプロトコルが開始されたことをユーザーに通知します。

2.  **フェーズ変更のテストカバレッジの確保:**
    -   **ステップ 2.1: フェーズ範囲の特定:** このフェーズで変更されたファイルを特定するために、まず開始点を見つけます。`plan.md` を読み、前のフェーズのチェックポイントの Git コミット SHA を見つけます。前のチェックポイントが存在しない場合、範囲は最初のコミット以降のすべての変更です。
    -   **ステップ 2.2: 変更されたファイルのリスト:** `git diff --name-only <previous_checkpoint_sha> HEAD` を実行して、このフェーズ中に変更されたすべてのファイルの正確なリストを取得します。
    -   **ステップ 2.3: テストの検証と作成:** リスト内の各ファイルについて：
        -   **重要:** 拡張子を確認し、コード以外のファイル（例：`.json`, `.md`, `.yaml`）を除外します。
        -   残りの各コードファイルについて、対応するテストファイルが存在することを確認します。
        -   テストファイルが不足している場合は、作成 **しなければなりません**。テストを記述する前に、まずリポジトリ内の他のテストファイルを分析して、正しい命名規則とテストスタイルを決定します。

3.  **プロアクティブなデバッグを伴う自動テストの実行:**
    -   実行前に、テストを実行するために使用する正確なシェルコマンドをアナウンスする必要があります。
    -   **例:** 「フェーズを検証するために自動テストスイートを実行します。**コマンド:** `CI=true npm test`」
    -   アナウンスされたコマンドを実行します。
    -   テストが失敗した場合は、ユーザーに通知してデバッグを開始します。修正の提案は **最大2回** まで試行できます。2回目の提案された修正の後もテストが失敗し続ける場合は、**停止** し、永続的な失敗を報告してユーザーにガイダンスを求めてください。

4.  **詳細で実行可能な手動検証計画の提案:**
    -   **重要:** 計画を生成するために、まず `product.md`, `product-guidelines.md`, および `plan.md` を分析して、完了したフェーズのユーザー向け目標を決定します。
    -   必要なコマンドや具体的で期待される結果を含む、ユーザーを検証プロセスに導くステップバイステップの計画を生成する必要があります。

5.  **明示的なユーザーフィードバックの待機:**
    -   詳細な計画を提示した後、ユーザーに確認を求めます。「**これは期待通りですか？ yes で確認するか、変更が必要な点についてフィードバックをください。**」
    -   **一時停止** し、ユーザーの応答を待ちます。明示的な yes または確認なしに進まないでください。

6.  **チェックポイントコミットの作成:**
    -   すべての変更をステージングします。このステップで変更が発生しなかった場合は、空のコミットを進めます。
    -   明確で簡潔なメッセージとともにコミットを実行します（例：`conductor(checkpoint): Checkpoint end of Phase X`）。

7.  **Git Notes を使用した監査可能な検証レポートの添付:**
    -   **ステップ 7.1: ノート内容のドラフト:** 自動テストコマンド、手動検証ステップ、およびユーザーの確認を含む詳細な検証レポートを作成します。
    -   **ステップ 7.2: ノートの添付:** `git notes` コマンドと前のステップのフルコミットハッシュを使用して、検証レポートをチェックポイントコミットに添付します。

8.  **フェーズチェックポイント SHA の取得と記録:**
    -   **ステップ 8.1: コミットハッシュの取得:** 作成されたばかりのチェックポイントコミットのハッシュを取得します (`git log -1 --format="%H"`)。
    -   **ステップ 8.2: 計画の更新:** `plan.md` を読み、完了したフェーズの見出しを見つけ、`[checkpoint: <sha>]` の形式でコミットハッシュの最初の7文字を追記します。
    -   **ステップ 8.3: 計画の書き込み:** 更新された内容を `plan.md` に書き戻します。

9. **計画更新のコミット:**
    - **アクション:** 変更された `plan.md` ファイルをステージングします。
    - **アクション:** `conductor(plan): Mark phase '<PHASE NAME>' as complete` という形式のメッセージとともにこの変更をコミットします。

10.  **完了のアナウンス:** フェーズが完了し、チェックポイントが作成されたことをユーザーに通知します。

### 品質ゲート

タスクを完了としてマークする前に、以下を確認してください：

- [ ] すべてのテストが合格している
- [ ] コードカバレッジが要件を満たしている (80%以上)
- [ ] コードがプロジェクトのコードスタイルガイドに従っている
- [ ] すべての公開関数/メソッドが文書化されている (例: JSDoc)
- [ ] 型安全性が強制されている (TypeScript の型など)
- [ ] リンターや静的解析のエラーがない
- [ ] モバイルで正しく動作する (該当する場合)
- [ ] 必要に応じてドキュメントが更新されている
- [ ] セキュリティの脆弱性が導入されていない

## 開発コマンド

### セットアップ
```bash
npm install
```

### 日常の開発
```bash
npm run dev
npm test
npm run lint
```

### コミットの前に
```bash
npm run build:cli
npm test
```

## テスト要件

### ユニットテスト
- すべてのモジュールに対応するテストが必要です。
- 適切なテストセットアップ/クリーンアップメカニズム (例: beforeEach/afterEach) を使用してください。
- 外部の依存関係をモックします。
- 成功と失敗の両方のケースをテストします。

### 統合テスト
- 完全なユーザーフローをテストします。
- データベース（Redis）のトランザクションを検証します。
- 認証と認可をテストします。

## コードレビュープロセス

### セルフレビューチェックリスト
レビューを依頼する前に：

1. **機能性**
   - 機能が指定通りに動作する
   - エッジケースが処理されている
   - エラーメッセージがユーザーフレンドリーである

2. **コード品質**
   - スタイルガイドに従っている
   - DRY 原則が適用されている
   - 明確な変数/関数名
   - 適切なコメント

3. **テスト**
   - ユニットテストが包括的である
   - 統合テストが合格する
   - カバレッジが十分である (80%以上)

## 定義の完了 (Definition of Done)

以下の条件を満たしたときにタスクは完了します：

1. すべてのコードが仕様通りに実装されている
2. ユニットテストが記述され、合格している
3. コードカバレッジがプロジェクトの要件を満たしている
4. ドキュメントが完了している（該当する場合）
5. コードがすべての設定されたリンターおよび静的解析チェックに合格している
6. 実装ノートが `plan.md` に追加されている
7. 適切なメッセージとともに変更がコミットされている
8. タスクサマリーを含む Git note がコミットに添付されている
